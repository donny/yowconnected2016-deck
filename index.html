Continuous Integration ♻️

... a practice that requires developers
to integrate code into a shared repository.

Each check-in is then verified by an automated build.

Continuous Integration (CI) is a development practice .. usually several times a day.
The automated build check in allows us to detect problems early and locate them more easily.

Benefits

  • Increase visibility

  • Reduce integration risk

  • Automate build and deployment

Benefits of CI, among others:
It increases the visibilit of bugs and errors.
It reduces the risk of integrating code changes.
It facilitates automated build and faster deployment.

Problems & Issues 🔥

👻

The previous CI infrastructure served our needs at that time.
Jenkins was reasonably okay and it fit with our requirements then.
But we had several issues. And it longer met our requirements now.

Issues

  • Upgrading Jenkins

  • Upgrading OS X and Xcode

  • Unintuitive Configuration

The upgrading process was complicated and it needed a few Wiki pages.
The build pipeline needed constant care and it became very brittle with successive updates.
As we put more updates and changes done by multiple people over the years.
The infrastructure became like a vase that needed to be handled carefully.
And we were afraid of installing plugins just in case it breaks Jenkins.

🐶  🐱

Or, in other words, we were treating our CI infrastructure like pets.

📢

Further, as a side effect, it affected our workflow.
Failed tests or nightly builds were not noticeable.
Code commits and test builds were not tightly integrated.

We did a spike on various CI infrastructure early this year.
We settled on Buildkite.
It meets our needs and it's really awesome.
It's also used by other tribes at REA and it has been vetted by the security team.
Let's compare Buildkite with other systems.
If you look at the market of CI systems, there are two types.
Managed and unmanaged providers.
The managed providers are closed system and multi tenant.
Sometimes, they have on-premises offering.
In contrast to unmanaged provider.
Talk about the components: UI / master / scheduler vs agents.
Buildkite is different.
The UI / master or the scheduling component is managed.
The agents are managed by us.
The agent code is open source and we can be confident that our source code doesn't leave our system.
This is what Buildkite looks like.
We can see the pipeline and the builds with three green builds and two red builds.
If we click on one of the builds, we will see the detailed view.
Where we can see the build pipeline consisting of four build steps: Plan Upload, Setup, Unit Test, Upload
One of the strengths of Buildkite is that we can represent the build pipeline as code.
Usually as a YAML file. And what we see now was our config.yml file.
The file statically defined our build pipeline.
Where we can see how the build steps correspond to the sections in the YAML file.
But about two months ago, we switched to dynamic pipeline.

Dynamic Pipeline

  • Branches (e.g. master, production)

  • Commit messages (e.g. XCODE8)

Rather than having a static file to define our build steps.
We use Buildkite dynamic pipeline feature to generate build steps dynamically based on conditions.

osx-vm-factory

This is a set of scripts and Packer templates to create OS X VM for VirtualBox and VMware.
The factory has its own Buildkite pipeline.
And the workflow of the VM factory is best described through the build steps.
And I'll explain the build steps with the next few slides.

In the pipeline upload step, we simply upload and setup the Buildkite CI steps.

  • Set necessary variables

  • Ensure software is installed

  • Ensure folders are created

  • Get OS X from App Store

  • Build OS X disk image

  • Pack base OS X image and save

  • Get base OS X image

  • Build OS X + Xcode disk image

  • Pack OS X + Xcode image and save

  • Get OS X + Xcode image

  • Build OS X + Xcode + Tools disk image

  • Pack OS X + Xcode + Tools and save

If needed...
Example of tools: ruby gems, carthage, swift lint, etc.
These automated steps save us time and generally once built, each step takes only about 10 seconds.
